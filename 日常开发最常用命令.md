## 🚀 项目初始化

### 首次设置（只做一次）
```bash
# 1. 配置Git用户信息
git config --global user.name "你的用户名"
git config --global user.email "你的邮箱"

# 2. 生成SSH密钥
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519 -N ""

# 3. 复制公钥到GitHub
cat ~/.ssh/id_ed25519.pub | clip
# 然后在GitHub的Settings → SSH and GPG keys中添加

# 4. 测试连接
ssh -T git@github.com
```

### 新项目初始化
```bash
# 1. 创建项目目录
cd "C:\Users\Eldridge\Desktop\my-project"
mkdir my-project && cd my-project

# 2. 初始化Git仓库
git init

# 3. 添加文件并提交
git add .
git commit -m "feat: 初始化项目"

# 4. 关联GitHub仓库
git remote add origin git@github.com:用户名/仓库名.git

# 5. 首次推送
git push -u origin master
```

## 💼 日常开发流程

### 基本三步曲（90%的时间都在用）
```bash
# 1. 添加修改到暂存区
git add .

# 2. 提交代码
git commit -m "简洁描述做了什么"

# 3. 推送到远程
git push

# 快捷方式（一行命令完成三步）
git add . && git commit -m "描述" && git push
```

### 查看状态
```bash
# 查看当前工作目录状态
git status

# 查看未暂存的修改
git diff

# 查看已暂存的修改
git diff --staged

# 查看提交历史
git log --oneline

# 查看文件历史
git log 文件名
```

## 🌿 功能开发流程

### 创建功能分支
```bash
# 创建并切换到功能分支
git checkout -b feature/功能名称

# 或者先创建，再切换
git branch feature/功能名称
git checkout feature/功能名称
```

### 在功能分支上开发
```bash
# 查看当前分支
git branch

# 开发并提交
git add .
git commit -m "feat: 实现XX功能"
git commit -m "fix: 修复XX问题"  # 可多次提交

# 查看当前分支和远程分支的关系
git branch -v
```

### 推送功能分支到远程
```bash
# 首次推送（设置上游分支）
git push -u origin feature/功能名称

# 后续推送（已设置上游）
git push
```

## 🔍 代码审查与合并

### 在GitHub上合并（推荐）
1. 打开GitHub仓库页面
2. 点击 "Pull requests" → "New pull request"
3. 选择要合并的分支：
   - base: `master` (目标分支)
   - compare: `feature/功能名称` (源分支)
4. 查看代码差异
5. 填写标题和描述
6. 点击 "Create pull request"
7. 点击 "Merge pull request"
8. （可选）删除已合并的分支

### 命令行合并
```bash
# 切换到master分支
git checkout master

# 拉取最新代码
git pull origin master

# 合并功能分支
git merge feature/功能名称

# 推送到远程
git push

# 删除已合并的本地分支
git branch -d feature/功能名称

# 删除远程分支
git push origin --delete feature/功能名称
```

## 🔄 同步远程更新

### 方法一：直接拉取合并（推荐）
```bash
# 简单直接，一条命令完成
git pull origin master

# 针对特定分支
git pull origin feature/分支名
```

### 方法二：分步拉取合并
```bash
# 1. 获取远程更新但不合并
git fetch origin master

# 2. 查看远程更新内容
git log --oneline origin/master
git diff master..origin/master

# 3. 确认要合并后手动执行
git merge origin/master
```

### 处理合并冲突
```bash
# 1. 查看冲突文件
git status

# 2. 手动编辑冲突文件
# 解决冲突后，保存文件

# 3. 标记冲突已解决
git add 冲突文件名

# 4. 提交合并
git commit -m "解决冲突，合并更新"

# 5. 推送到远程
git push
```

## 🚨 紧急处理命令

### 撤销本地修改
```bash
# 撤销工作区的修改（未git add）
git checkout -- 文件名

# 撤销暂存区的修改（已git add但未commit）
git restore --staged 文件名
git checkout -- 文件名

# 撤销提交（未push）
git reset HEAD~1          # 保留修改在暂存区
git reset --hard HEAD~1   # 删除所有修改
```

### 强制推送（谨慎使用）
```bash
# 强制推送（会覆盖远程代码）
git push -f origin master

# 推送所有分支
git push --all
```

### 切换分支
```bash
# 切换到已有分支
git checkout master

# 切换并创建新分支
git checkout -b feature/新功能

# 切换到上一个分支
git checkout -
```

## 📝 完整工作流程示例

### 示例1：日常修复Bug
```bash
# 1. 查看当前状态
git status

# 2. 修复bug
# 编辑文件...

git add .

# 4. 推送
```

```bash
git checkout -b feature/用户管理

# 2. 开发功能
# 编辑代码...
git add .

# git add .
```

```bash
# 1. 查看远程更新

# 3. 如有冲突，解决冲突后
git add .
git commit -m "解决冲突，同步更新"

```

## 💡 最佳实践

1. **频繁提交**：开发过程中频繁小提交，而不是一次性大提交
2. **清晰的提交信息**：使用 `type: 描述` 格式
   - `feat:` 新功能
   - `fix:` 修复bug
   - `docs:` 文档更新
   - `style:` 代码格式化
3. **分支命名规范**：`feature/功能名`、`fix/问题描述`
4. **及时推送**：完成开发后及时推送，避免丢失代码
5. **定期同步**：开始工作前先拉取最新代码

## ❓ 常见问题解答

**Q: push到哪里了？**
A: 推送到当前所在的分支，不是master分支

**Q: 不想push时怎么保存？**
A: 只需要 `git add` 和 `git commit`，不运行 `git push`

**Q: 如何知道当前在哪个分支？**
A: `git status` 或 `git branch`，当前分支前面有*号

**Q: 合并后为什么文件还在？**
A: 合并成功后，建议使用 `git branch -d` 删除已合并的分支
